clan-rewrite
YAML-driven NE Configuration

Replacing 10K lines of hardcoded per-NE C with declarative YAML + Lua — add a network-element type by dropping in a file, no recompile.

Dan Nuzzo  ·  dnuzzo@lightriver.com
Design approved 2026-05-27 · revised 2026-06-04
Working name: ncland (New CLAN Daemon)

1The Problem

cnc/sdi/src/clan.c — ~10,000 lines — is the CLI gateway between netFLEX clients and network elements.

Every per-NE behavior is hardcoded in C:

  • Prompt regexes, login sequences, keepalive commands
  • Paging handling, post-command processing, disconnect cleanup
  • Spread across #defines, switch ladders, and per-vendor setup<Vendor>Connection() functions

Adding a new NE type = editing C + recompiling + redeploying.

1Goals

  • Drop-in NE types — a .yaml (+ optional .lua) in a config dir. No C recompile.
  • Declarative data in YAML — all the static per-NE config.
  • Imperative logic in Lua — complex, conditional per-NE behavior (login).
  • Coexists with clan.c — dispatches per NE type; cutover is per-NE and reversible.
  • Provable parity — behavior verified against legacy before any NE migrates.
  • Shared patterns with ncland — same transport, eventing, process model.

3What Varies Per NE (the motivation)

Catalogued from clan.c — ~25 NE-type families, each with its own dtype:

  • Prompt detection — a primary/secondary regex pair per NE
  • Login — ~15 families just need a newline after connect; others have multi-step vendor dialogues
  • Keepalives — per-NE command + interval (whoami\n/600s, \n/43s, exec ping %h\n/600s)
  • Paging — three more-prompt families (Cisco/Juniper, Ciena/Fuji, default)
  • Disconnect — some need clean save (TSS5 write-memory, WaveServer config save)
  • Transport — telnet/ssh mix, varying banner & command timeouts

There is no central per-NE table today. This rewrite extracts one.

4Architecture — Warehouse / Worker

Warehouse (1 per daemon)

  • Owns the ZMQ ROUTER socket
  • Owns the YAML registry
  • Owns the nfdb event SUB socket
  • Single-threaded epoll loop

Workers (N, e.g. -w 4)

  • Each owns a partition of NE-side I/O
  • Talk libssh / telnet to NEs
  • Run the YAML stepper + Lua VM

Boundary discipline

ZeroMQ touches only the external caller boundary. Warehouse↔worker IPC is a Unix-domain ctl_sock carrying tagged frames ('F' fd-handoff via SCM_RIGHTS, 'C' cmd, 'R' rsp, 'X' shutdown).

4.6Four Planes

1 · Registry

Startup map: ne_type → (parsed YAML, compiled regexes, Lua module).

2 · Stepper

Runs the YAML step language for post_command / disconnect_steps. Per-session state, inside workers.

3 · Lua bridge

Exposes session + global tables to scripts. Drives login.

4 · Transport + eventing

ROUTER for clients · nfdb SUB for NE lifecycle · ctl_sock to workers.

│   ├── fuji_1finity_s100.yaml   # extends: fuji_1finity_x100.base
│   └── ...

Conventions: filename matches ne_type: lowercased · _defaults.yaml merged under every file · inherit one parent via extends: · abstract bases group shared behavior.

7The NE YAML

ne_type: CIENA_RLS              # must match clan.c dtype enum
extends: ciena.base             # optional: inherit a parent
display_name: "Ciena RLS"
vendor: ciena
schema_version: 1

transport:
  protocol: [ssh, telnet]       # list = preferred order, fallback
  banner_timeout_s: 30
prompt:
  primary_regex:  '[-a-zA-Z0-9/:^ #]+#'
  need_newline_after_connect: true
keepalive:  { enabled: true, interval_s: 600, command: "exec ping %h\n" }
paging:     { more_pattern: '--More-- or (q)uit', more_response: " " }
post_command:   [...]           # declarative step language
disconnect_steps: [...]
lua_module: ciena_rls.lua       # REQUIRED — exposes login()

A minimal file is ~6 lines — or 2–3 when it extends: a base.

7.5Inheritance — extends: + abstract bases

Merge order

_defaults.yaml
   → [ extends chain, root first ]
      → this file   (wins)
  • Maps deep-merge, derived wins
  • Lists replace wholesale
  • Multi-level, single parent
  • abstract: true = template only, never registered

Example: Fuji X100 family

fuji_1finity_x100.base.yaml  abstract
  ├ s100.yaml  extends → S100 dtype
  ├ t100.yaml  extends → T100 dtype
  └ l1xx.yaml  extends → L1XX dtype

Three real dtypes, behavior written once. Generalizes to vendor bases (ciena.base) and transport bases.

8–9Step Language & Lua Bridge

Declarative steps

For post_command / disconnect_steps — bodies are mostly data.

Ops: expect · send · expect_send · branch · label/goto · set · lua

Paging drains automatically; step budget guards runaway loops.

Login lives in Lua

function M.login(session, global, args)
  -- banner detection, credential
  -- branching, prompt capture …
end

Login dialogues are inherently imperative — the step language was awkward for them. session API: send/expect/get_var/fail.

Reserved session vars (bridge-populated): user · password · p_enable · p_read · p_write · card_type · host · ne_type · slot · neid

11NE-Lifecycle Eventing — the nfdb bus

One ingestion path for seed and runtime: a ZMQ SUB socket on the nfdb pub/sub broker (lives in niimxd).

 nclan-seed ──PUBLISH app.ne.seed──┐
 (reads frame_link, startup)       ├─► nfdb XPUB ─► clan SUB ─► dispatch ─► open/close
 ne-DB writes ──db.ne.add/update/  ┘   nfdb_pub.sock           (app.ne. + db.ne.)
   delete (runtime, auto)
TopicAction
app.ne.seed / db.ne.addOpen if dtype owned (idempotent)
db.ne.updateClose + reopen (re-applies any change)
db.ne.deleteClose the session

Events are two frames [topic][yaml], identity only (neid, dtype, ip, tid, slot) — no credentials on the bus.

4.4 · 11.5Seeding — nclan-seed

  • The warehouse never reads frame_link. A standalone CLI does.
  • nclan-seed walks frame_link and PUBLISHes one app.ne.seed per NE
  • The warehouse is seeded by consuming those events — same code path as runtime
  • Startup order: subscribe first, then fork/exec the seeder (pub/sub is non-durable)
  • Idempotent & re-runnable — duplicate/already-open = no-op; re-seed = just re-run it

One seeder run feeds every consumer (clan-rewrite, ncland, …). Each picks the dtypes it owns.

10.1Warehouse Startup

1  parse CLI flags
2  install signalfd (SIGTERM/SIGHUP) → epoll
3  scan ne/*.yaml — resolve extends, detect cycles
4  per file: merge chain → validate → compile regexes → check lua_module
5  build the registry  (ne_type → entry)
6  bind ZMQ ROUTER, register ZMQ_FD in epoll
7  connect nfdb SUB (app.ne. + db.ne.), register fd     ← subscribe first
8  fork N workers (socketpair ctl_sock each)
9  fork/exec nclan-seed → app.ne.seed events arrive on the SUB
10 enter the epoll loop
11 log: loaded=N types, workers=M, nfdb=ok|degraded

13Validation & Cutover

Proving parity

  • Static CI — YAML validates, regexes compile, ne_type↔dtype round-trip
  • Replay — capture live clan.c byte traces; replay against rewrite (≥20 clean to graduate)
  • Shadow — both backends run; responses diffed daily; 7 clean days to promote

Per-NE state machine

legacy-only
   → shadow
      → primary
         → legacy-removed

Reversible at every step. Last step is a per-NE PR deleting the C code. Never rushed.

14Open Questions

Summary

Config, not code.

Per-NE behavior becomes data (YAML) + scripts (Lua). New NE types ship without touching C. Cutover is per-NE and reversible. Seeding and runtime lifecycle flow through one event path.
